# AnyShare Family 7 知识助手技术原理解读及最佳实践

## 最佳实践

> 提示：本章节介绍的最佳实践内容均以AnyShare 7.0.6.4版本环境为演示示例，具体操作时请以实际环境为准。

## 1. 问答智能体最佳实践

问答智能体作为知识助手的关键应用形态，通过结构化 FAQ 匹配与非结构化文档检索相结合的方式，实现高效精准的问答服务。

### 1.1 工作流程

问答智能体采用双路知识检索机制，其知识源包括【加盟 AI 智能管家 FAQ】知识仓库和【加盟商知识资料公共文件夹】文档库。具体工作流程如下：

1. **FAQ 优先匹配**：当用户提问后，系统首先在【加盟 AI 智能管家 FAQ】知识库中进行精准匹配，将用户问题与FAQ 知识的标题（主问）及相似问题进行匹配。若匹配成功，则直接返回预设的标准答案。

2. **文档检索兜底**：若 FAQ 匹配未命中，则系统自动在【加盟商知识资料公共文件夹】的文档库，通过文档检索召回相关内容，并基于大模型生成式问答技术产出答案。

**匹配流程**：

![FAQ匹配流程图](../images/diagram_page104_img4.png)

该流程设计既保证了高频问题的响应速度与答案一致性，又通过文档检索扩展了知识覆盖范围，确保各类问题都能得到有效解答。

**应用实例说明**：

在【加盟 AI 智能管家 FAQ】知识仓库中配置有一条典型 FAQ 知识，其中：
- 主问（标题）："什么是电子发票"
- 相似问："怎么给顾客开发票"

当用户提问"什么是电子发票"或"怎么给顾客开发票"时，系统精准匹配到该FAQ知识，直接返回标准答案，并在引用资料中明确标注信息来源。

而对于"AMP是指哪些费用"这类未在【加盟 AI 智能管家 FAQ】中预设的问题，系统自动转入文档检索流程，基于【加盟商知识资料公共文件夹】中的相关文档内容生成回答，展现智能体的泛化问答能力。

### 1.2 FAQ 匹配实现和调优

在问答智能体的工作流程中，FAQ匹配作为首轮检索环节，其准确性直接影响后续文档检索的触发条件与最终回答质量。

#### 1.2.1 匹配配置方案概述

在初始配置下，系统存在 5 例因语义理解错误导致的错误匹配（即本应进入文档库召回的问题，被错误匹配至不相关的 FAQ）。为解决这几个问题，实施了以下关键调优措施：

**1. 阈值参数调优**
- **去掉关键字检索**：默认关键字检索采用"OR 匹配"逻辑，易引入噪音，造成召回内容不精致，关闭后可提升召回精准度与系统性能。
- **提高向量召回阈值**：将向量召回阈值（min_score）设置为 0.9，小于这个分数的结果将会被过滤掉，去掉低相关性结果，减少噪音干扰。
- **提高重排序阈值**：将重排序环节的相似阈值设置为 0.9，小于这个分数的结果会被过滤，实现多层级的噪音削减。

**2. 核心名词切片过滤**

经阈值参数调优处理后，仍存在1例因语义高度相似导致的错误匹配。具体而言，当用户查询与FAQ库中的某些条目在语义上高度接近但存在关键信息差异时，仅依靠阈值调整难以实现有效区分。

为此引入核心名词过滤机制，通过提取用户查询中的核心名词（如"配送费"），并对召回结果进行校验过滤，即可有效排除不含该核心名词的相似切片。

该机制通过以下流程实现：
- 使用LLM模块精准识别查询中的核心实体与关键限定词
- 将核心名词作为必要过滤条件，对候选FAQ切片进行筛选
- 仅保留同时满足语义相似度要求且包含核心名词的匹配结果

#### 1.2.2 基于LLM的核心名词提取

**核心名词提取提示词示例**：

#### 1.2.4 双重匹配机制

采用关键字与语义双重匹配策略，基于LLM块将用户问题与FAQ主问/相似问进行匹配：
- 仅当关键字和语义均匹配成功时，返回对应FAQ ID
- 任一条件不满足则返回空字典

**FAQ匹配提示词示例**：

```
/prompt/

你是一个专业的FAQ（常见问题）匹配系统。你的任务是接收一个`用户查询 (User Query)`，并将其与`FAQ数据库 (FAQ Database)`进行关键字和含义双重一致性匹配。如果找到关键字和含义双重一致性匹配的FAQ条目，你将返回其对应的FAQ条目的`id`；否则，你将告知用户未能找到匹配。

**详细规则与要求：**

1. **输入格式：**
   - `"用户查询"` (string): 用户的原始问题。
   - `"FAQ数据库"` (array of object): 包含多个FAQ条目。

2. **匹配判据：**
   - **首先关键字匹配：** 对于`用户查询`，提取所有显式关键字和关键短语，基于提取的关键字和`主问`、`相似问`的关键字进行匹配并返回候选结果。

### 1.3 文档检索实现和调优

在FAQ匹配环节未命中时，系统将转入文档检索流程。

#### 1.3.1 文档检索配置方案

文档检索调优主要围绕提升召回准确率、降低大模型幻觉、保证回答准确性和答案输出稳定性三个目标展开：

**1. 阈值参数调优**

- **优化两路召回处理逻辑**：将处理逻辑调整为取两路召回切片的并集后再进入重排序环节，避免有效信息被误过滤。
- **提高重排序阈值**：将重排序（Rerank）相似阈值提高至 0.2。
- **控制生成随机性**：将大模型 temperature 参数从默认值 1 调整为 0，确保相同查询下输出答案的一致性。

**2. 提示词工程调优**

为降低大模型幻觉、提升问答准确性，对提示词进行针对性优化，例如明确要求模型必须基于参考文档原文进行回答。

#### 1.3.2 文档召回执行逻辑

**文档召回代码示例**：

min_score = 0.2  # rerank 中的分数阈值

async def f1_main(as_ctx: dict, query: str) -> Union[list, str]:
    """
    关键字向量双路召回+权重过滤
    """
    ranges = ["12D62C2B910E4F47B57DB748371D18B2"]
    ranges = [await itemIdToDocId(x) for x in ranges]
    embeddings = await embedding(query)

    data = {
        "limit": top_n,
        "text": query,
        "embedding": embeddings,
        "ranges": ranges,
        "item_output_type": ["doc"],
    }

    search_res = await slice_search(data)
    dense_results = search_res["doc"].get("dense_results", [])
    sparse_results = search_res["doc"].get("sparse_results", [])

    res = merge_result(dense_results, sparse_results)
    return res

def merge_result(dense_results: list, sparse_results: list) -> list:
    """融合结果，并集"""
    id_map = set([x["slice_id"] for x in dense_results])
    res = dense_results
    for v in sparse_results:
        if v["slice_id"] in id_map:
            continue
        id_map.add(v["slice_id"])
        res.append(v)
    return res
```

### 1.4 智能体性能优化实践

#### 1.6.1 同步函数异步化

为提升系统并发处理能力，对关键函数进行异步化改造：

**同义词查询函数优化**：将同义词替换后的 new_query 函数改造为异步模式，显著提升查询响应速度。

**FAQ答案提取函数优化**：对FAQ匹配中的正文答案提取函数进行异步化重构，避免I/O阻塞。

#### 1.6.2 LLM块输出模式优化

**文档检索LLM块优化**：从专家模式调整为Basic模式，支持答案内容的实时流式返回。

**相似问生成LLM块优化**：同样调整为Basic模式以支持流式输出。

#### 1.6.3 流式输出阻塞问题解决

LLM块后的函数会等待LLM块执行完毕后才会进行执行，这将阻塞LLM块的流式返回，请务必去除此类函数块。

## 2. 图谱增强（Graph RAG）问答

### 2.1 图谱增强问答流程

图谱增强型（Graph Retrieval-Augmented Generation）知识问答是一种结合知识图谱结构化知识与文档切片精准检索的智能问答方案。该技术通过图谱关系召回候选文档集，并基于文档内容进行切片级检索，从而为复杂查询提供更精准、可靠的答案。

**图谱增强问答流程**：

![图谱增强问答流程图](../images/diagram_page210_img5.png)

- **前期知识化准备工作**：对文档进行元数据处理，构建文档图谱，自动提取文档编目和标签，内容索引和向量化。

- **在线问答增强流程**：
  1. 利用知识图谱召回候选文档集，对拼接后的文档编目和标签重排打分
  2. 取得分Top N（可自定义N的大小数量）相关文档，交由大模型组织并返回答案
  3. 在返回答案中明确标注回答的参考来源（文档）

#### 3.1.1 课程讲解评估Agent配置方案

课程讲解评估Agent是通过用户给定的课程讲解内容，从「参考答案库」中自动匹配对应的参考答案，并根据参考答案对讲解内容进行评估，最终生成评估报告。

**Agent输入**：
- query（用户问题，比如：帮我生成一份课程讲解评估报告）
- as_ctx（传递过来的用户身份验证等信息）
- user_doc_content（用户输入的课程讲解内容全文）

**核心处理流程**：

**1. 参考答案召回**：基于输入的文本内容，从数据源中召回候选文档集。

**2. 获取召回文档object_id**：获取召回文档集中所有文档的object_id。

**3. 提取召回文档的全文**：获取召回文档的完整文本内容。

**4. 自动匹配参考答案**：基于用户输入的课程讲解内容，自动从所有召回结果中匹配最相关的文档作为参考答案。

**参考答案匹配提示词**：

```
# 角色及适用场景
你是一个高度专业化的文档检索与匹配专家。你的唯一任务是分析用户上传的文档，并精准地找出内容最相关、主题最匹配的文档。

# 任务指令
从以下文档中选择符合要求的文档

待选择文档：
{{retrievers_block_content}}

用户问题：
{{query}}

参考内容
{{user_doc_content}}

回答要求：
1、解析参考文档：全面理解用户文档的核心主题、关键论点、涉及的具体实体等信息。
2、从待选择文档中推荐与参考内容相匹配合适的文档，如果待选择文档中没有文档，则输出"未找到参考答案"
3、推荐文档的排序按照版本排序，版本越新，越靠前。
```

**5. 生成课程讲解评估报告**：

**课程评估报告生成提示词**：

```
# 角色及适用场景
你是一位经验丰富的教育学专家和资深课程评审员。你的任务是对比分析用户上传的文档和标准的参考答案，并从内容完整性、表达逻辑性、教学有效性等维度进行专业、公正的评估，最终生成一份内容翔实、具有建设性的评估报告，以html格式呈现。

# 任务指令
用户的参考内容：
{{user_doc_content}}

标准答案：
{{reference_answer}}

以标准答案{{reference_answer}}为参考依据，从以下四个核心维度对用户的参考内容{{user_doc_content}}进行逐项评估：

1、关键点覆盖度 (40分)：
   仔细解析参考答案中所有核心论点、关键步骤、重要概念和必备知识点。

2、语言逻辑与通顺度 (30分)：
   评估讲解的条理性、语言的流畅性、连贯性。

3、案例引用与实证支持 (20分)：
   检查用户是否使用了具体案例、事例、数据或故事来阐释抽象概念或复杂理论。

4、讲解结构与吸引力 (10分)：
   评估开场是否吸引人？结尾是否有总结和升华？整体节奏是否得当？
```

## 4. 核心概念

| 核心概念 | 定义 |
|---------|------|
| **检索增强生成（RAG）** | 检索增强生成（Retrieval-Augmented Generation）是一种将信息检索与大语言模型生成能力相结合的技术框架。通过在生成答案前先从知识库中检索相关文档作为上下文，显著提升大模型回答的准确性和可信度。 |
| **RAG Pipeline** | 指检索增强生成（RAG）的端到端处理流程，涵盖从文档解析、切片向量化、语义检索到答案生成的完整链路。 |
| **知识助手** | 知识助手是 AnyShare 为了实现企业级智能知识管理而专门开发的内置助手，采用自然语言问答式交互体验。 |
| **智能体（Data Agent）** | 用户可自主创建的面向特定业务场景的智能化功能单元。通过配置人设、回复逻辑和数据源，组合模型设置、技能调用、数据源选择、数据流编排及错误调试等能力。 |
| **大语言模型（LLM）** | 基于海量文本训练的自然语言处理模型，具备强大的文本理解、生成和推理能力。 |
| **知识图谱** | 以"实体-关系-实体"三元组形式组织和存储知识的语义网络，能够直观表达业务概念间的复杂关联。 |
| **图谱增强（Graph RAG）** | 图谱增强生成（Graph Retrieval-Augmented Generation），一种融合知识图谱的增强检索机制。 |
| **Dolphin-Lang** | 爱数自主研发专为智能体设计的流程配置语言。支持使用自然语言的方式通过判断、探索等多种复杂模式，对智能体数据流进行编排。 |
| **Agentic Workflow** | 智能体驱动的工作流程，通过多个 Agent 协同与调度，完成复杂多步骤任务的处理与决策。 |
