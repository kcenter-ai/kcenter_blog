# AnyShare Family 7 知识助手技术原理解读 - 知识化与图谱构建

## 3.2.1.4 内容结构树提取

在完成元数据处理或基础文本提取后，系统可进一步对多模态文档进行深度解析，识别其内部逻辑结构（如章节、表格、图表等）。该环节依赖于自建Python 算法库，能够生成具备语义层次的内容结构树，为后续的向量化切片、知识图谱构建等高阶知识化处理提供高质量的结构化数据基础。

对于含有复杂版式、嵌套表格、列表或特定结构的文档，通用引擎可能无法保留基于文档内容的完整语义结构。AnyShare 通过集成自研 Python 算法库，提供针对不同格式文档内容（章节、段落、表格、标题等逻辑结构）的精准识别能力，并构建包含层次关系的文档内容结构树。

**文件格式处理说明：**

| 格式 | 处理方式 |
|------|----------|
| Excel (.xlsx) | 采用openpyx 库处理，提供完整的读写能力和格式控制功能 |
| Excel (.xls) | 结合使用xlrd 进行读取和xlwt 进行写入操作 |
| PowerPoint | 采用python-pptx 库实现文件解析和内容提取 |
| Word (.docx) | 使用python-docx 库读取和写入，能够准确识别文档中的段落、表格、图片等元素 |
| Word (.doc) | 使用 ASPOSE 转换组件将其转换为新版格式后再进行处理 |
| PDF | 采用 PyMuPDF 工具作为基础解析引擎，针对复杂格式引入版面分析模型进行精细化处理 |
| PDF 扫描件 | 采用 OCR 模型进行内容识别，结合图像预处理技术提升识别准确率 |
| HTML | 采用 Scrapy 框架实现网络文档的抓取和解析，通过预设规则和机器学习模型去除广告、导航栏等无关内容 |

## 3.2.2 文档分类及管理机制

在实际应用场景中，同一文档常被多次请求解析。若每次均完整执行解析流程，将导致不必要的资源消耗与响应延迟。为优化系统资源利用并提升响应速度，产品设计了基于"主文档-副文档"架构的文档分类与管理机制。

### 3.2.2.1 主文档-副文档

**主文档**：指用户直接上传到 AnyShare 的原始文档，如 DOCX、PDF、图像及音视频文件等，作为知识化处理的源头和依据。

**副文档**：由主文档经过转换、转码、分析、识别等处理过程所产生的辅助文档。例如：
- 由 DOCX 转换生成的 PDF 预览文件
- 从视频中提取的音频字幕
- 经 OCR 识别后输出的结构化文本等

副文档作为中间处理结果，其核心价值在于支持处理结果的多场景复用，避免重复计算，提升系统效率。

**副文档生命周期**：指副文档在系统中的存活管理策略，旨在平衡存储成本与访问性能。

### 3.2.2.2 主副文档管理机制

副文档由主文档结合特定处理参数共同生成，同一主文档可能因调用参数不同衍生多个副文档。副文档通常设有可配置的过期时间策略（如 7 天），并支持根据最后访问时间自动续期。一旦主文档被删除，其衍生的所有副文档将同步清理。

**副文档类型一览表：**

| 类型 | 数量 | 有效期 | 支持文件格式 | 前置类型 | 生成参数 |
|------|------|--------|--------------|----------|----------|
| 加密 | 1个 | 跟随主文档 | - | 主文档 | - |
| 解密 | 1个 | 跟随主文档 | - | 主文档 | - |
| 纯文本提取 | 1个 | 跟随主文档 | Word, Excel, PPT... | 主文档 | - |
| 摘要提取 | 1个 | 跟随主文档 | Word, Excel, PPT... | 主文档 | - |
| 带时间戳纯文本 | 1个 | 跟随主文档 | 3gp, mp4, rmvb, flv, mkv, aac, mp3, ogg... | 主文档 | - |
| OCR | 1个 | 跟随主文档 | Jpg, png, bmp, gif... | 主文档、内联图片副文档 | - |
| 结构化纯文本 | 1个 | 7天 | Word, Excel, PPT... | 主文档 | - |
| 缩略图 | 1个 | 7天 | Word, Excel, PPT... | 主文档 | - |
| 外发包 | 1个 | 7天 | - | 主文档 | 密码、过期时间、打开次数 |
| 转PDF | 多个 | 7天 | Word, Excel, PPT... | 主文档 | 输出一页，像素 |
| 文档内嵌图片提取 | 多个 | 7天 | Word, Excel, PPT... | 主文档 | 图片最小宽度和高度 |
| 水印 | 多个 | 7天 | Jpg, png, pdf, gif, bmp... | 主文档、PDF副文档 | 布局、旋转角度、文字、间距 |
| 解压缩 | - | 7天 | zip, rar, 7z, tar, tar.gz | 主文档 | - |

## 3.3 文档内容切片，构建细粒度知识单元

![文档切片处理流程图](../images/diagram_page21_img4.png)

在完成多模态文档解析与结构化提取之后，为提升知识检索的准确性与语义针对性，需对文档数据进行细粒度切片处理，以便将已解析的文档内容按照预设规则划分为具备独立语义的片段，并将其存入向量数据库以供后续检索使用。

切片策略的设定对知识服务质量具有关键影响：
- 若切片过大，包含信息过多，可能导致语义分散，影响检索精度
- 若切片过小，则易丢失关键上下文，削弱语义完整性

### 3.3.1 基于文档层次结构的切片策略

AnyShare 主要采用基于文档版面分析结果的智能切片方式，能够识别文档中段落、标题、图表等元素的边界与位置关系，保持其语言连贯性与语义完整性。

此外，AnyShare 还集成基于语义相似度的动态切分能力，通过计算文本片段间的语义相关性，在语义发生显著变化的位置进行切分。对于缺乏明确层级结构的文档，系统支持基于固定长度（size）和分隔符（如句号、换行）进行切分。

### 3.3.2 自定义分片配置

为满足不同业务场景的需求，系统支持对切片参数进行灵活配置，主要包括：

- **切片大小（size）**：设置文本片段的长度阈值
- **分隔符**：指定用于切分的标点符号或特殊字符（如换行符、句号等）

> **备注**：直到 AnyShare 7.0.6.6 版本，产品依旧不支持切片自定义的支持，会在 AnyShare 7.0.6.7 版本开始对全文提取策略和切片向量策略进行配置显性化和自定义配置支持。

## 3.4 文档内容向量化，完成语义空间知识映射

在完成文档内容的细粒度切片之后，系统进入知识化流程的关键步骤——向量化（Embedding）。作为大语言模型（LLM）检索增强生成（RAG）的核心技术之一，向量化通过嵌入模型（Embedding Model）将文本、图像等非结构化数据转换为高维向量空间中的数值表示，为后续语义检索与智能问答建立基础。

### 3.4.1 向量化技术优势

与传统关键词匹配技术相比，向量化技术在解决语义理解不准确、低效检索等方面展现出显著优势：

| 特性 | 传统关键词检索（BM25） | 向量检索 |
|------|------------------------|----------|
| 匹配方式 | 字面匹配，无法理解语义 | 语义相似度比较 |
| 语义理解 | 无法理解"AI"和"Artificial Intelligence"的关联 | 可直接找到表述不同但语义相关的内容 |
| 跨模态检索 | 难以直接关联文本与图像 | 可将异构数据映射至统一向量空间 |
| 大规模检索 | 直接基于原始文档检索效率极低 | 通过高维向量相似度计算快速定位 |

### 3.4.2 向量化原理与流程

向量化的核心在于利用深度神经网络模型学习文本的深层语义特征：

1. **嵌入模型加载**：采用预训练的嵌入模型，这些模型基于海量语料完成训练，能够精准捕捉语义信息

2. **文本预处理**：对切片后的文本进行清洗和标准化，以符合嵌入模型的输入要求

3. **向量计算**：将预处理后的文本输入嵌入模型，模型通过内部复杂网络结构，最终输出一个固定的浮点数向量。这个向量即为文本在语义空间中的"坐标"

4. **向量存储与索引**：计算得到的向量被存入专用的向量数据库（如 OpenSearch）中，并构建索引，以支持后续相似度搜索

### 3.4.3 向量化应用流程

向量化技术的价值在后续的检索与生成流程中得以体现：

1. **查询向量化**：当用户发起查询时，系统将用户查询语句转换为与存储向量同一空间的一个查询向量

2. **语义召回**：在向量数据库中，通过计算余弦相似度等度量方式，快速查找与查询向量最相似的文档切片向量

3. **结果排序与返回**：根据相似度得分对结果进行排序，并按照Top-K 规则返回最相关的向量集合

4. **上下文构建与答案生成**：通过向量ID 映射原始文本片段，将这些片段作为上下文与用户问题共同组合成提示（Prompt），输入大语言模型以生成最终答案

### 3.4.4 多模态文件向量化流程

AnyShare 针对不同类型文件（Office 文件、音视频文件、图片文件）的向量化处理流程：

#### 处理流程概述

1. **文档类型识别（mime-type）**：用户上传文件后，DocSet 借助 Tika 读取文件前 4K 文件头数据，识别文件的真正类型，判断该文件是否为系统支持处理的文件类型

2. **任务编排与下发（任务链生成）**：DocSet 根据系统状态（如是否安装加解密服务）以及识别出的文件类型，编排构建对应的任务节点，并将这些任务节点写入数据库。后台将按照任务优先级和顺序下发到 ContentPipeline 进行执行

#### 同类型文件的具体处理

**Office 文件处理流程：**
- 若系统安装了加解密服务，先对文件进行解密操作
- 根据"CPU 版切片服务开关"是否打开，进入不同的文档内容提取分支流程：
  - **开关打开时**：进行全文提取、内联图片提取、OCR 识别、全结构化文本提取等操作
  - **开关关闭时**：直接进行全文提取
- 提取的内容生成 slice_vector（docset）（切片向量），经 intelli_extractor（智能提取器）处理后，交由 CPU 切片服务或 GPU 切片服务后续处理

**音视频文件处理流程：**
- 若系统安装了加解密服务，先对文件进行解密操作
- 进行带时序的语音提取，生成 slice_vector（docset），经 intelli_extractor（智能提取器）处理后，交由 GPU 切片服务或 CPU 切片服务后续处理

**图片文件处理流程：**
- 若系统安装了加解密服务，先对文件进行解密操作
- 进行 OCR 识别，生成 slice_vector（docset），经 intelli_extractor（智能提取器）处理后，交由 GPU 切片服务或 CPU 切片服务后续处理

**生命周期控制：** 当所有任务节点执行完成后，会生成一系列相关副文档，由 DocSet 按照指定的过期时间对这些副文档进行管理，超期的将会被删除以释放资源，而频繁被访问的副文档则会更新其过期时间，避免重复计算

> **备注**：公司现有 GPU 版的切片服务，支持 PDF 扫描件等一些额外的格式，但是尚处于性能不稳定阶段，资源占用较多，如果有需要，可以联系研发咨询。

## 3.5 知识图谱构建，还原业务逻辑关系

![知识图谱构建架构图](../images/diagram_page38_img4.png)

知识图谱是一种结构化的知识表示框架，其核心价值不仅在于对事实数据和实体属性的存储，更在于对事物间关系、上下文逻辑及语义关联的深度捕捉。通过构建实体（如"项目文档""部门文件夹"）、关系（如"隶属于""引用了"）及属性（如"创建人""更新时间"）所组成的多维关系网络，深度刻画事物之间的语义关联、上下文逻辑及依赖关系。

**知识图谱与向量数据库协同架构：**
- **知识图谱**（依托图数据库进行关系查询）：负责结构化关系推理
- **向量数据库**：处理语义相似性匹配

系统响应用户查询时，先通过图数据库定位核心实体及其关系路径，再通过向量数据库检索相关文档片段，最终合并结果作为上下文输入大模型。

### 3.5.1 AnyShare 知识图谱

![AnyShare知识图谱架构](../images/diagram_page41_img5.png)

AnyShare 知识图谱构建基于 AnyDATA Framework 3 实现，依托其实体识别、知识抽取能力，从文档元数据和内容中提取关键信息，形成面向企业业务场景的领域知识图谱。

#### 3.5.1.1 内置文档图谱

系统提供预定义的内置文档图谱，基于通用文档元数据自动构建实体关系网络，支持基础文档关联查询与知识展示等应用。

**图谱构建与更新机制：**
知识图谱的构建及更新依赖于系统对文档中心文档状态变化的实时响应，通过事件驱动的工作流机制实现。当文档中心发生文件上传、移动、删除或复制等操作时，系统通过内置工作流监听相应事件，并自动触发相应的元数据提取与图谱更新流程。

#### 3.5.1.2 自定义知识图谱

为适应特定业务场景需求，系统支持在默认图谱基础上构建自定义知识图谱。相较于标准的 AnyShare 文档图谱，自定义知识图谱可基于文档内容提取的文档摘要、文档编目和文档标签、文档标题，能够尽可能地从图谱获取到最相关的文档对象。

**图谱构建流程：**
1. **定义知识图谱名称**：基于业务场景为图谱设置识别名称
2. **配置数据源**：指定一个或多个 AnyShare 文档库作为数据来源
3. **定义/导入本体**：可以在 AnyShare 文档图谱的基础上，新增业务专属的实体和关系类型
4. **建立属性映射**：维护图谱实体/关系属性和数据源属性的映射关系，将数据源中的各个字段和实体或关系的属性对应起来

> **提示**：自定义图谱时，需与业务场景核心对象深度沟通确认，确保各项定义贴合业务场景。

#### 3.5.1.3 图谱数据的手动管理及维护

进入 AnyDATA 图谱详情页，您可以搜索、查询并手动更新维护图谱数据。

### 3.5.2 知识图谱开放能力

系统提供完整的图谱开放接口，支持：

1. 支持图谱数据（三元组 `<实体，关系，实体>`）批量插入接口
2. 支持通过定义图分析查询应用实现专用的图数据分析目的
3. 支持对图谱进行更新后，重新全量抽取相应数据进行图谱重新构建
4. 支持通过图谱搜索接口自定义对接搜索应用

## 3.6 索引及存储，支撑高效检索的底层数据架构

在完成文档解析、切片及向量化等系列知识化处理后，系统需将处理结果持久化存储并构建高效索引，以支撑上层智能应用。针对不同类型的数据特征，AnyShare 采用差异化的存储策略，实现对元数据、文本内容、切片向量的统一存储管理。

### 3.6.1 元数据存储

文件上传完成后，索引构建服务通过监听消息事件触发元数据获取流程。该服务调用文档引擎接口，提取文档的基本元数据（如文档名称、格式、上传时间、大小、存储路径等）与扩展元数据（包括标签、编目和摘要等）。

- **基本元数据**：文档的基础描述信息
- **扩展元数据**：对文档内容特征的补充描述

两类元数据共同构成文档的完整描述体系，最终由索引构建服务统一写入 OpenSearch 的元数据索引库，为后续检索提供基础属性筛选能力。

### 3.6.2 文本内容存储

元数据写入后，索引构建服务向文档集服务提交异步文本提取任务，以获取文件完整文本内容。文档集服务接收任务后，会启动文本提取流程，将根据文件格式调用相应解析工具（如 Apache Tika、OCR 服务等），提取文档纯文本内容。

提取结果返回后，索引构建服务将其与对应元数据关联，并存储于 OpenSearch 的同一索引库中，形成支持全文检索的内容索引。

### 3.6.3 切片向量存储

全文内容存储至 OpenSearch 后，索引构建服务向向量提取服务提交异步任务，任务内容为对文档文本进行切片处理并转换为向量。向量提取服务接收到任务后，按照预设规则将文档文本切分为多个独立切片，再通过向量模型将每个切片转换为对应的向量数据。

**切片记录结构：**
每个切片在 OpenSearch 向量索引库中是一条独立记录，包含：
- 基本元数据信息
- 切片的文本内容（原始文本片段）
- 切片的向量内容（转换后的向量数据）

**设计优势**：向量索引库与元数据、内容索引库为独立索引库，这种设计避免不同类型数据存储相互干扰，同时优化向量检索性能。

### 3.6.4 索引构建及存储

索引构建服务获取到需建立索引的文件后，启动索引构建流程。具体步骤如下：

#### 1. 文本分词处理

获取文档的完整文本内容后，首先进行分词操作：
- **中文**：按语义分词
- **英文**：按空格分词

随后对分词结果进行清洗：
- 去除标点符号
- 过滤停用词（Stop Word）

**停用词示例：**
- 中文："的"、"是"、"在"
- 英文："the"、"and"、"is"

过滤后可有效减少索引规模，提升存储与检索效率。

#### 2. 字典创建与排序

分词清洗后得到的有效词汇称为"词项"（Term）。索引组件将所有"词项"整合为一个词汇字典，并按照字母顺序排序，确保"词项"的有序性与快速定位。

#### 3. 倒排链表构建

索引组件基于排序后的词汇字典，为每个词项创建对应的文档倒排链表（Posting List）：
- 倒排链表中记录包含该 Term 的所有文档标识（如文档 ID）
- 同时记录"Term"在文档中的位置、出现频次等信息
- 最终将所有"Term"的倒排链表与词汇字典关联，形成完整的文本检索索引
- 存储于 OpenSearch 内容索引库中，为后续全文检索提供支持
