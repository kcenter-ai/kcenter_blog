# AnyShare Family 7 知识助手技术原理解读及最佳实践

## 智能体：知识驱动的问答与决策

### 1. 智能体概述（Data Agent）

#### 1.1 智能体架构

智能体是知识驱动的问答与决策核心，其架构包含以下核心组件：

**核心工作流程：**
1. 接收查询请求（query）与技能指令
2. 意图识别
3. 调用技能/数据召回
4. 主控总结
5. 返回结果
6. 结果高亮与引用处理
7. 最终实现页面的流式展现

#### 1.2 核心概念解释

**1. 知识助手**

知识助手是 AnyShare 为实现企业级智能知识管理而专门开发的 AI 助手。它采用自然语言问答式交互，围绕知识问答、搜索、撰写、审阅核心能力，提供 AI 原生的人机交互窗口。知识助手内置集成了多种知识管理和运营能力，包括内容总结，翻译，帮我写作，文档问答和文件查找等核心技能。

**2. 智能体（Data Agent）**

为更好服务客户业务场景，AnyShare 支持用户自主创建面向特定业务场景下的智能体（Data Agent）。通过预设人设及回复逻辑，配置编排（包括模型设置、技能调用、数据源选择和数据流启用）以及可视化调试，同时依托 AnyShare 承载的企业知识，结合知识图谱、大模型能力，智能体可专注于特定场景的知识应用（如法务合同审阅智能体、官网咨询智能体），为各类型业务场景提供智能化能力支撑。

**术语背景：**
AnyShare 知识助手始于 7.0.5.0 版本的"AnyShare 认知助手"，随后在版本迭代中不断演进：
- 7.0.5.0：AnyShare 认知助手
- 7.0.6.3：升级为"AnyShare 内容机器人"
- 7.0.6.4：更名为"AnyShare AI助手"
- 7.0.6.5：确立为"AnyShare 知识助手"

**3. LLM（大型语言模型）**

LLM 是基于海量文本数据训练的人工智能模型，能够理解、生成和处理人类语言。用户通过精心设计的提示词工程，可驱动 LLM 完成特定的 AI 操作，是实现智能体核心能力的技术基础。

**4. Agent**

Agent 是能够感知环境，进行自主决策并执行动作以实现特定目标的智能实体，可以作为技能被智能体（Data Agent）调用。主要包括两类：

- **主控Agent**：从 AnyShare 7.0.6.4 版本开始，在 AnyShare 中创建的每个智能体都采用技能编排模板。这代表每个智能体都有一个主控 Agent 负责协调所有其他 Agent，整合最终结果并返回。在 AnyDATA Agent Facotry 中，命名包含 "Control_Agent"的均为主控 Agent。

  > 注意：若想在AnyShare智能体中正确显示主控Agent的返回结果，需注意其返回结构。

- **技能Agent**：在技能编排模板中，除主控 Agent 外，还存在诸多具有专用目的的技能 Agent，他们被主控 Agent 调用以协助完成特定任务。技能 Agent 模板类型多样，包括问答模板（可召回数据）、空白模板（可自定义内部逻辑）等。

  > 注意：若想在主控Agent中正确显示技能Agent的返回结果，需注意其返回结构。

**5. Tool**

Tool 是被设计用于执行特定任务或有限功能的被动资源/器具，本身不具备自主决策能力，必须在 Agent 或用户创建的智能体（Data Agent）调用时，才会执行预设的明确操作。

### 2. 智能体创建（标准创建逻辑、智能创建逻辑）

#### 2.1 创建流程概述

AnyShare支持所有用户自主创建智能体（Data Agent），以满足各类智能应用场景中不同用户的个性化需求。其创建过程支持全流程智能化，通过提供用于描述所需智能体的自然语言信息或配置项提示词，即可借助大模型自动生成其创建的配置参数，随后可基于可视化调试页面先进行效果测试，再保存/发布。

**AnyShare对智能体（Data Agent）实行细粒度权限管控：**
- **创建权限上**：仅管理员可以创建API调用类助手，为系统提供智能化支撑的同时，保障接口调用的规范性与服务可靠性。
- **发布权限上**：仅白名单用户可发布组织范围内的助手，无权限用户创建的助手仅供个人使用，确保公共助手质量可控，同时保护个人助手隐私。

#### 2.2 支持创建的智能体类型

| 类型 | 说明 |
|------|------|
| **适用于用户** | 指 AnyShare 界面上所有可直接使用的智能体，普通用户和管理员都有权创建此类智能体。创建并发布后，适用范围内的用户可以使用此智能体。 |
| **适用于API调用** | 指基于API接口为 AnyShare 其他系统功能（例如"扩写"、"续写"等功能）的实现提供服务的智能体，此类智能体仅管理员有权限创建。创建并发布后，可通过API调用该智能体。 |

#### 2.3 创建模式

支持以下两种创建模式：

| 模式 | 说明 |
|------|------|
| **智能创建** | 使用自然语言一句话描述所需智能体（功能、特点、应用场景等），系统将基于此自动生成名称、简介、人设及回复逻辑、开场白、预设问题等智能体创建参数，支持根据实际场景进一步调整创建参数，以降低智能体配置门槛，帮助快速创建。 |
| **标准创建** | 从零开始或基于已有模版创建智能体。 |

#### 2.4 创建参数说明

| 参数 | 说明 |
|------|------|
| **名称** | 智能体名称。最大 50 个字符，不允许包含空格，特殊字符只能包含-或_。 |
| **简介** | 一句话描述智能体用途，方便使用者知晓智能体作用。最大 255 个字符。 |
| **人设及回复逻辑** | 智能体在对话中扮演的角色、职责、回复风格，以及技能使用规则，支持手动填写、智能生成两种配置方式。自动生成时，支持自行调整提示词，以生成所需配置参数，若需使用技能，可通过"{}"引用技能名称。 |
| **模型及参数** | 支持选择智能体生成回答时所依赖的大语言模型。当前支持DeepSeek-R1、DeepSeek-V3、Qwen、GPT4o、自研Tome等主流大语言模型。支持配置以下模型参数： |
| **技能** | 支持选择 Agent 或工具（tool）作为智能体技能，以处理更复杂的任务。最多允许添加 50 个。 |
| **数据** | 智能体数据配置区，支持配置数据源、临时区、数据流等参数。 |
| **对话开场白** | 支持固定开场白和个性化开场白两类配置。 |
| **预设问题** | 展示在智能体问答交互窗口的上方，用于抛砖引玉引导用户提问、介绍作答规范等。 |
| **预览与调试** | 保存后，创建者可对智能体效果进行测试及调试。调试界面将完整展示知识助手从输入到输出的执行过程。 |

#### 2.5 模型参数配置

| 参数 | 说明 | 默认值 | 可设范围 |
|------|------|--------|----------|
| **随机性 (temperature)** | 用于控制大模型生成文本的随机性和创造性。较高值会使输出更多样、不可预测，较低值会使输出更遵循指令要求。 | 0.0 | 0.0-2.0 |
| **采样样本 (top_p)** | 动态选择概率累积超过p的最小词集，从中随机采样，用于控制大模型生成文本的多样性。较高值会扩大采样候选词范围，增加生成文本的多样性但相关性可能降低；较低值仅从最可能的少量词汇中选择，输出更保守，但生成内容更准确。 | 1.0 | 0.0-1.0 |
| **单词回复限制 (max_tokens)** | 用于控制模型输出的Tokens长度上限（通常100 Tokens约等于 150 个中文汉字）。较高值可能会使模型生成过长回答，较低值可能导致模型生成的回答截断，浪费计算资源。 | 3000 | 10-32000 |
| **前K采样 (top_k)** | 用于控制候选词数量，模型将从概率最高的前k个词中随机选择。较高值可能会使模型输出更多样，但可能会引入不相关词；较低值会使模型输出更保守。 | 1 | 1-1000 |
| **话题新鲜度（presence_penalty）** | 用于控制模型在生成文本时引入新话题的倾向。较高值会鼓励模型切换至新话题，使内容更具多样性；较低值则允许模型围绕当前话题深入讨论，使内容更加集中，但可能缺乏变化；值为0，则模型将根据自身概率自由决定是否重复话题。 | 0 | -2-2 |
| **频率惩罚度（frequency_penalty）** | 降低已生成文本中高频词汇的重复出现。较高值会促使模型避免重复用词，增加词汇的多样性；较低值则允许词汇的重复使用；值为0，则模型将根据自身概率自由选择词汇。 | 0 | -1-2 |

#### 2.6 标准创建逻辑

### 3. 文档召回

应用阶段，智能体（Data Agent）将根据用户查询从海量数据中快速、准确地定位文档相关信息，进行文档召回。文档召回是基于OpenSearch搜索能力实现的核心功能。

#### 3.1 关键字召回（Lexical Search）

关键字召回是基于词项匹配实现的高效检索，这是搜索引擎最传统、最基础的能力，适用基于精确关键词或固定表述进行查询的场景。

**工作原理：**

1. **文本处理**：对用户查询语句（Query）和文档（Document）进行分词（Tokenization），将其拆分成一个个独立的词项（Term）。例如，"OpenSearch 的向量功能"可能被分词为["opensearch"，"向量"，"功能"]。

2. **构建倒排索引**：OpenSearch 会为所有文档建立一个倒排索引（Inverted Index），这个映射表记录了每个词项出现的文档、位置及频率信息。类似书本末尾的索引页。

3. **匹配与评分**：根据查询中的词项，在倒排索引中快速定位包含这些词项的文档集合。再使用 BM25 等排名函数计算相关性分。综合考虑词频（TF）、逆文档频率（IDF）和字段长度等因素。

**优势与局限：**

| 优势 | 局限 |
|------|------|
| 擅长处理精确关键词、数字、代码或专有名词等字面匹配场景（如产品型号"iPhone 15 Pro"、"错误码404"），计算效率高 | 效果高度依赖于分词质量，无法理解语义，难以处理同义词（如"电脑"和"计算机"）、多样化表述、错别字等问题 |

#### 3.2 向量召回（Vector Search / Semantic Search）

向量召回基于语义理解进行检索，其工作原理如下：

1. **向量化（Embedding）处理**：采用预训练的深度学习模型（如BERT、Sentence-BERT等）将查询语句和文档内容转换为高维向量表示。这些向量在数学空间中捕捉文本的深层语义特征，语义相近的文本其向量表示在空间中的距离也更接近。

2. **向量索引构建**：将文档向量存储至OpenSearch并构建高效的向量索引结构（如HNSW或IVF-PQ）。这些专用索引通过近似最近邻算法（ANN），在保证检索精度的同时，显著提升了高维空间中的搜索效率。

3. **相似度匹配计算**：用户查询时，系统将查询内容转换为向量，并在预先构建的向量索引中执行相似度搜索。常见的相似度度量方式包括余弦相似度（CosSim）、内积（Inner Product）和欧氏距离（L2 Distance）等。

**优势与局限：**

| 优势 | 局限 |
|------|------|
| 深刻理解语义，即便查询和文档没有共享相同关键词，只要语义相近就能被召回。能够有效应对同义词、多样化表述、甚至错别字等问题 | 计算资源消耗更大，通常需要 GPU 等硬件加速向量化过程。在处理精确术语、代号或数字时，效果可能不如关键字召回直接 |

#### 3.3 混合检索（Hybrid Search）

单一召回策略在应对复杂搜索场景时均存在局限性，为兼顾关键字检索的精准性与向量检索的语义理解能力，系统采用混合检索架构。

**实现混合检索的核心挑战：**
两种召回机制的分数体系存在本质差异：关键字召回采用的BM25分数理论无上限，而向量相似度得分通常限定在[0,1]区间。

**统一排序流程：**

![混合检索架构图](../images/diagram_page51_img5.png)

1. **多路并行召回**：系统同步执行关键字查询和向量查询，获得两份独立的候选文档列表；
2. **分数归一化处理**：采用min-max归一化、L2归一化或RRF等算法，将两类分数统一映射至可比较的数值区间；
3. **加权融合排序**：为归一化后的关键字分数与向量分数分配不同的权重（如 0.4 和 0.6），通过加权求和方法计算每篇文档的最终相关性得分；
4. **（可选）重排序（Rerank）**：系统支持将混合检索产出的Top-K结果送入精细化重排序模型（如基于BERT的Cross-Encoder）进行二次精排。

#### 3.4 召回优化策略

针对三种主要文档类型设计了差异化的召回优化策略：

| 文档类型 | 说明 |
|----------|------|
| **Doc** | 包含office、pdf、音视频、图片等类型。当前支持关键字召回与向量召回，可根据应用端传入的参数进行返回。召回接口支持传入text或embedding参数，如果传入text参数则返回关键字召回的切片列表（稀疏召回），如果传入embedding参数则返回向量召回的切片列表（稠密召回）。通过ranges参数指定目标文档库范围，根据用户访问权限对召回的文件进行过滤。 |
| **WIKIDoc** | 召回逻辑与Doc文档召回基本一致，但是其范围参数为data_set_id，而非ranges。 |
| **FAQ** | 存储的是问题、问题相似问法以及问题的答案。在混合召回模式下会返回完整FAQ条目，而不是对应的切片，即每个FAQ返回时都会包含问题、相似问法和答案。 |

**切片上下文补全：**

在文档召回过程中，系统返回的切片通常是与查询直接相关的片段，这些片段在原文中可能并不连续。为解决这一问题，系统会在初始召回结果的基础上执行上下文补全操作。对于每一个被召回的切片，系统会自动检索其在原始文档中相邻的前后内容，将前n个切片与后n个切片一并纳入上下文范围。当前系统默认配置为获取下文2个切片，上文0个切片。

#### 3.5 重排序（Rerank）策略

在完成初步召回后，引入了重排序环节对候选结果进行精细化处理。召回与重排序共同构成从"海选"到"精选"的完整流程。

重排序模型通过深度语义分析，逐一对候选切片与用户查询进行深层相关性计算，输出更精确的匹配分数（如mge_score）。由于该模型结构相对复杂、计算开销较大，需特别注意部署性能优化。

经重排序处理后，系统将获得按语义相关度重新排列的切片列表。基于mge_score的分布特征，可设定rerank的分值阈值对结果进行进一步筛选，有效过滤低相关性内容。

#### 3.6 可配置参数

| 配置类别 | 参数 | 说明 |
|----------|------|------|
| **查询优化配置** | 智能改写和查询扩充功能开关 | 控制是否启用 AI 对用户原始查询提问进行语义改写与扩展优化 |
| **多路召回配置** | 关键字召回功能开关 | 是否启用基于关键词匹配的传统检索 |
| | 相似度召回功能开关 | 是否启用基于语义相似度的向量检索 |
| | 相似度阈值 | 设定向量匹配的最低分数要求 |
| **上下文管理** | 召回上下文块数据量设置 | 控制单次召回返回的上下文切片数量 |
| **召回块设置** | 召回和重排序参数配置 | 可根据业务场景特性、计算资源约束与性能要求进行组合调整 |

### 4. 图谱召回

![图谱召回架构图](../images/diagram_page166_img4.png)

在基于关键词和向量的文档召回基础上，系统进一步提供基于知识图谱的语义召回能力。图谱召回通过挖掘三元组（实体、关系与属性）之间的深层语义关联，为复杂查询和推理类问题提供更精准的结构化信息支撑。

#### 4.1 图谱召回原理

图谱召回是基于知识图谱的检索方式，通过查询关键字或语义匹配从知识图谱中召回相关的三元组信息（实体-关系-实体）或子图数据。

**核心流程：**

1. **多路召回**：同时通过实体名称、实体属性和实体关系，从知识图谱中召回与查询最相关的实体集合。

2. **子图扩展与获取**：图谱召回过程首先在实体属性层面并行执行关键字召回与向量召回，筛选出与查询意图相关的候选实体集合。基于这些实体，系统采用双向一度扩展策略查询其关联子图，生成包含完整语义信息的三元组列表。每个三元组的结构化表示为：
   - 实体A及其属性
   - 实体B及其属性
   - 关系C（实体A与实体B之间的关系）及其关系属性

3. **文本化输出**：召回的三元组内容默认拼接为纯文本格式，作为大语言模型的输入上下文，为问答生成提供结构化的知识支撑。

#### 4.2 召回配置参数与调优策略

图谱召回支持高级配置，用户可根据具体场景调整以下参数：

| 配置类别 | 参数 | 说明 |
|----------|------|------|
| **实体识别及分词配置** | 大模型选择 | 配置用于实体识别和查询理解的大模型 |
| | 文本匹配实体数量 | 设置基于文本匹配召回的实体数量上限 |
| | 向量匹配实体数量 | 设置基于向量相似度召回的实体数量上限 |
| **结果精排配置** | 重排序保留数量 | 精排后保留的参考信息数量 |
| | 图谱BGE相似度阈值 | 基于BGE模型的重排序相似度过滤阈值 |
| | 长文本上限 | 输出上下文的最大长度限制 |

通过这些配置参数的组合调优，系统能够更精准地筛选出与用户查询高度相关的图谱信息，有效提升复杂问答的准确性和推理能力。

### 5. 图谱增强

在基础检索能力之上，知识图谱增强机制通过结构化知识与非结构化文档的协同检索，能够提升复杂问答的准确性与推理深度。

#### 5.1 基于数据源的增强问答方案

当前知识库问答是基于数据源进行问答，默认是从已经解析的文档向量库（opensearch）中召回和 query 最相关的切片，将切片内容作为参考资料，将用户问题作为 query，送入 LLM 块进行问答，并基于 query 和 answer 送入相似问题 LLM 块生成相似问题，得到答案 answer 和 relastions_query，再增量流式返回给 Web 端渲染回答：
- 基于 answer 渲染回答
- 基于 relations_query 渲染相关追问问题
- 基于切片所属文档显示参考文档
- 再基于 answer 和切片内容的相似度显示引用资料

#### 5.2 基于图谱与文档库的增强问答方案

**方案一：多路召回内容融合**

在标准方案的基础上，执行图谱召回（三元组集合）与文档库召回（文件切片）：
- 通过图谱召回获取与查询语义最相关的子图信息（三元组集合）
- 将图谱返回的三元组数据与文档库召回和重排的切片内容直接拼接，作为大模型的增强上下文输入

**方案二：实体导向的切片增强**

在前一方案的基础上，进一步利用图谱的实体引导能力：
- 通过图谱召回获取与查询语义最相关的子图信息（三元组集合），并提取三元组中的文档实体列表
- 以文档实体列表为切片范围，从文档库中召回与查询最相关切片（集合A），此过程需要添加自定义函数
- 将集合A与标准召回切片（集合B）通过交并集策略拼凑
- 经重排序后，作为大语言模型的参考内容输入，确保答案兼具广度与准确性

**方案三：图谱引导的两阶段召回**

采用完全由图谱驱动的召回策略，即图谱召回（文件列表）+ 在已给定的文件列表中进行切片召回：
- 通过图谱召回获取和查询最相关的子图信息（三元组集合）
- 提取三元组中的文档实体列表作为切片范围，从文档库中召回与查询最相关的切片（集合A），此过程需要添加自定义函数
- 对集合A进行重排序后，直接作为LLM模块的参考内容输入，确保答案与知识图谱的高度一致性

#### 5.3 增强方案对比

| 方案 | 核心要点 | 差异点 |
|------|----------|--------|
| 方案一 | 图谱三元组 + 文档切片 | 相较于标准方案，增加了query相关文档的元数据 |
| 方案二 | 图谱切片 + 文档切片 | 相较于标准方案，增加了以query相关的图谱三元组文档范围的切片内容 |
| 方案三 | 图谱三元组 + 图谱切片 | 相较于标准方案，仅通过图谱完成切片召回 |
| 标准方案 | 文档切片 | 直接基于文档切片作为大模型输入，进行问答 |

### 6. Agent 核心概念

Agent作为承载业务智能化的最基本节点，是大模型能力表达的最小业务逻辑单元。通过对大模型对话能力的标准化封装，Agent形成可独立集成和复用的智能能力模块。

#### 6.1 Agent创建机制

当前有两种Agent创建入口：

**方式一：通过AnyShare智能体（Data Agent）创建**

在 AnyShare 中新建智能体(Data Agent)时，可直接创建并引用技能 Agent。目前仅支持直接引用基于以下两种标准模板创建的技能 Agent：
- **文档问答模板**：适用于基于特定数据源的召回问答场景
- **大模型知识问答模板**：适用于基于大模型通用知识的回答场景

**方式二：在AnyDATA中直接创建**

在AnyDATA中可直接新建技能Agent，支持选择所有类型的Agent创建模板。用户可以在 AnyShare智能体新建页面直接引用创建的Agent，也可以通过AnyDATA主控中通过Dolphin语法调用已创建的Agent。

#### 6.2 Agent调用方式

当前Agent支持三种主要业务使用场景：

| 场景 | 说明 |
|------|------|
| **智能体流程协作** | 作为智能体问答的关键流程，通过多个 Agent 协同，以完成特定的复杂业务流程 |
| **工作流嵌入节点** | 作为嵌入工作流的智能处理单元，为非结构化数据处理提供AI能力 |
| **独立功能模块** | 作为独立能力点直接被系统功能引用，如 Graphqa 找文件智能体 |

针对第一种场景，Agent提供两种显式调用方式：

1. **Dolphin语言调用**：通过类代码方式（@技能名）调用，例如：
   ```
   @web_search(query=$query,content=$content) -> skill_result
   ```

2. **自然语言调用**：在提示词中通过自然语言指令调用，例如：
   ```
   /explore/（system_prompt="调用技能（联网搜索）搜索和用户问题相关的资料，再回答用户问题"）
   /judge/（system_prompt="调用技能（联网搜索）搜索和用户问题相关的资料，再回答用户问题"）
   /prompt/请调用技能（联网搜索）回答用户问题
   ```

#### 6.3 LLM块调用方式

在AnyDATA中，每个Agent内的LLM块提供两种调用模式：

| 模式 | 说明 |
|------|------|
| **基础模式（Basic LLM）** | 适用于标准问答场景，直接输出大模型生成的文本内容（即大模型给出的回答） |
| **专家模式（Expert LLM）** | 支持通过Dolphin语法编写复杂业务流程，其输出为结构化JSON对象，适用于需要精确解析模型响应的场景 |

> 注意：专家模式下若需输出纯文本格式，则需额外解析JSON对象，此过程可能影响流式输出的实时性。

#### 6.4 函数块调用

Agent支持通过自定义Python函数块实现特定数据处理逻辑。函数块开发需遵循以下规范：
- 仅可使用系统内置依赖包
- 必须采用异步函数定义，避免阻塞整个Agent-app服务的运行
- 需充分考虑异常情况的发生及处理，防止因函数错误导致整个Agent-app的Python运行环境失效

#### 6.5 Agent数据召回

Agent召回块支持文档召回与图谱召回能力，召回结果将作为上下文参考文档输入后续LLM块。

若对现有召回策略不满意，或存有其他个性化召回需求，用户可通过自定义Python函数块调用以下接口：
- **slice_search**：执行文档切片召回
- **slice_fetch**：获取切片上下文信息
- **reranker**：调用Reranker模型

权限验证可通过Agent入口参数ax_ctx获取用户token进行设置。

### 7. 临时区

#### 7.1 功能概述

临时区功能支持用户在与智能体对话过程中上传本地文件或AnyShare内部文件进行解析，使 AI能够基于所上传文档内容进行问答、总结、翻译等操作。该功能有效扩展了知识助手的应用场景，使用户能够针对特定文件内容进行问答，大大提升AI应用的使用灵活性。

#### 7.2 参数配置说明

| 参数 | 说明 | 示例 |

#### 7.3 工作模式与原理

用户在临时区上传的文档后，系统自动将文档存储至AnyShare个人文档库的临时区目录。

> 注意：若未开通个人文档库，临时区功能将无法正常使用。

**文档内容提取机制：**
- 用户发起文件上传或者进行文件选择时，系统会调用结构化全文获取接口（/api/docset/v1/subdoc/full_structed_text），发起文档结构化文本提取任务（包含对内嵌图片的文本识别）
- 通过轮询机制，检查上传文档的结构化全文是否成功提取，完成后将文档添加至临时区

**文档内容处理策略：**

用户发起基于临时区文档的会话问答时，为适配AI模型对输入长度的限制（默认Token数为32k，以模型上下文会话Token大小的3/4为最大输入限制），系统提供以下动态文档内容处理方式：
- 若文档总长度在模型限制范围内，直接使用全部内容
- 若内容超长，系统提供两种处理机制：
  1. **简单截取模式**：截取文档前部内容至最大限制长度，并提示用户已截断的比例
  2. **文档段落筛选模式**：基于用户查询与文档结构化全文进行相似度计算，筛选最相关片段作为会话输入，并确保选择段落的总长度不超过输入长度限制

#### 7.4 文档内容筛选模式（相似度匹配）

文档内容筛选用于处理因文档过大而超出AI模型处理能力的情况。系统根据预设筛选策略，自动选取与用户问题最相关的内容片段。

**处理流程：**

1. **文档预处理**
   系统依据文档长度选择分块策略，并采用智能分句算法将文档切分为多个文本块，确保语义完整性。系统根据文档长度自适应调整分块大小：
   - 小型文档（＜1000字符）：分块单块大小为100字符
   - 中型文档（1000–5000字符）：分块单块大小为200字符
   - 大型文档（＞5000字符）：分块单块大小为300字符

2. **文档筛选流程**

![文档筛选流程图](../images/diagram_page103_img4.png)

   - 分词处理：对用户问题及文档内容进行中文分词
   - 相似度计算：基于Jaccard相似度算法计算文本块与问题的相关性
   - 逐级匹配：从高相似度阈值起，逐步放宽条件以筛选相关内容
   - 内容整合：将筛选出的最多10个相关段落组合为最终输入内容

3. **缺省处理**
   若文档内容筛选模式未能获取足够内容，系统将尝试段落级匹配；如仍不满足条件，则截取文档前5句作为默认内容

#### 7.5 内容整合与提示

系统将处理后的文档内容按指定格式整合，并通过提示信息，提醒用户若已上传相关文档，应优先基于所提供文档内容进行回答；若用户未明确要求使用其他功能，则不应主动调用额外技能。

最终，系统会将用户问题及相关文档内容一并提交至大语言模型，由模型生成基于文档上下文的针对性回答。

**提取后的结构化全文内容结构示例：**

```json
{
    "0": {
        "content": "第一页第一段，第一页第二段，第一页第三段，内嵌图片文本1，内嵌图片文本2",
        "page": 0,
        "seq": 0
    },
    "1": {
        "content": "第一页第一段",
        "page": 0,
        "seq": 1
    },
    "2": {
        "content": "第一页第二段",
        "page": 0,
        "seq": 2
    },
    "3": {
        "content": "内嵌图片文本1",
        "page": 0,
        "seq": 3
    },
    "4": {
        "content": "内嵌图片文本2",
        "page": 0,
        "seq": 4
    }
}
```

**说明：**
- 序号为0的段落包含全部的文档提取内容，包括已整合内嵌图片中的识别文本
- 序号1-4分别为后续语义切分形成的独立内容块
- page字段表示该内容块所在的页码
- seq字段表示该内容块在全文中的顺序编号（即全局序号，而非当前页面内的局部序号）

### 9. 智能体问答反馈与运营

在用户使用智能体完成问答交互后，系统通过完整的反馈与运营机制持续优化服务质量。该模块涵盖问答会话记录，以及用户收藏、点踩、点赞等反馈信息的数据统计。

#### 9.1 问答会话记录

系统通过历史记录机制保障对话连续性及数据可追溯性，主要处理逻辑包括：
- 支持获取历史对话上下文，维持当前会话中多轮对话的语义连贯
- 在对话结束时持久化存储完整对话记录
- 处理中断会话的自动合并情况
- 维护会话ID与QA ID的唯一性
- 记录对话长度、时间戳等关键元数据

#### 9.2 会话收藏机制

用户可将重要会话添加至收藏夹，系统通过在收藏表中的新增记录实现标记，并将相同会话（相同conversationID）下的历史记录设置为不删除状态（删除标记=1），确保持久保留。

#### 9.3 问答反馈收集（点踩/点赞）

系统通过点踩/点赞机制收集用户反馈：
- 反馈表通过detail_id（格式：bot_id/qa_id）关联具体问答内容
- 支持在回答生成完毕后，用户进行点踩/点赞
- 点踩时支持补充文字反馈与问题截图

#### 9.4 问答数据统计与分析

为实现对问答场景反馈信息的有效管理，AnyShare 基于多模块协作（页面、intelli-search、doc_center、DB等）实现用户反馈数据的有效收集、整合、存储，以及可视化统计展示。

| 角色 | 权限 |
|------|------|
| 普通用户 | 仅可查看自己发布智能体的统计数据 |
| 超级管理员 | 支持查看全局智能体运营分析数据 |

数据来源涵盖历史记录表与反馈信息表（对每个智能体进行分别统计）。

### 附录：索引及存储架构

在完成文档解析、切片及向量化等系列知识化处理后，系统需将处理结果持久化存储并构建高效索引，以支撑上层智能应用。针对不同类型的数据特征，AnyShare采用差异化的存储策略。

#### A.1 元数据存储

文件上传完成后，索引构建服务通过监听消息事件触发元数据获取流程。该服务调用文档引擎接口，提取文档的基本元数据（如文档名称、格式、上传时间、大小、存储路径等）与扩展元数据（包括标签、编目和摘要等）。

基本元数据是文档的基础描述信息，扩展元数据是对文档内容特征的补充描述，两类元数据共同构成文档的完整描述体系，最终由索引构建服务统一写入OpenSearch的元数据索引库。

#### A.2 文本内容存储

元数据写入后，索引构建服务向文档集服务提交异步文本提取任务。文档集服务接收任务后，会启动文本提取流程，将根据文件格式调用相应解析工具（如Apache Tika、OCR服务等），提取文档纯文本内容。

提取结果返回后，索引构建服务将其与对应元数据关联，并存储于OpenSearch的同一索引库中，形成支持全文检索的内容索引。

#### A.3 切片向量存储

全文内容存储至OpenSearch后，索引构建服务向向量提取服务提交异步任务，对文档文本进行切片处理并转换为向量。向量提取服务接收到任务后，按照预设规则将文档文本切分为多个独立切片，再通过向量模型将每个切片转换为对应的向量数据。

待切片与向量化处理完成后，每个切片包含两个核心信息：切片的文本内容（即切片对应的原始文本片段）和切片的向量内容。一个文件通常会被切分成多个切片，每个切片在OpenSearch向量索引库中是一条独立记录。

#### A.4 索引构建及存储

索引构建服务获取到需建立索引的文件后，启动索引构建流程：

1. **文本分词处理**：中文按语义分词、英文按空格分词，随后对分词结果进行清洗，去除标点符号并过滤停用词（如中文的"的""是""在"，英文的"the""and""is"等）

2. **字典创建与排序**：分词清洗后得到的有效词汇称为"词项"（Term）。索引组件将所有"词项"整合为一个词汇字典，并按照字母顺序排序

3. **倒排链表构建**：索引组件基于排序后的词汇字典，为每个词项创建对应的文档倒排链表（Posting List）。倒排链表中记录包含该Term的所有文档标识，以及Term在文档中的位置、出现频次等信息，最终形成完整的文本检索索引

### 附录：知识图谱

#### B.1 AnyShare知识图谱

**B.1.1 内置文档图谱**

系统提供预定义的内置文档图谱，基于通用文档元数据自动构建实体关系网络，支持基础文档关联查询与知识展示等应用。

**图谱构建与更新机制：**
知识图谱的构建及更新依赖于系统对文档中心文档状态变化的实时响应，通过事件驱动的工作流机制实现。当文档中心发生文件上传、移动、删除或复制等操作时，系统通过内置工作流监听相应事件，并自动触发相应的元数据提取与图谱更新流程。

**B.1.2 自定义知识图谱**

为适应特定业务场景需求，系统支持在默认图谱基础上构建自定义知识图谱。相较于标准的AnyShare文档图谱，自定义知识图谱可基于文档内容提取的文档摘要，文档编目和文档标签，文档标题能够尽可能地从图谱获取到最相关的文档对象。

**图谱构建流程：**
1. 定义知识图谱名称：基于业务场景为图谱设置识别名称
2. 配置数据源：指定一个或多个AnyShare文档库作为数据来源
3. 定义/导入本体：可以在AnyShare文档图谱的基础上，新增业务专属的实体和关系类型
4. 建立属性映射：维护图谱实体/关系属性和数据源属性的映射关系

> 提示：自定义图谱时，需与业务场景核心对象深度沟通确认，确保各项定义贴合业务场景。

**B.1.3 图谱数据的手动管理及维护**

进入AnyDATA图谱详情页，您可以搜索、查询并手动更新维护图谱数据。

#### B.2 知识图谱开放能力

系统提供完整的图谱开放接口，支持：
1. 支持图谱数据（三元组<实体，关系，实体>）批量插入接口
2. 支持通过定义图分析查询应用实现专用的图数据分析目的
3. 支持对图谱进行更新后，重新全量抽取相应数据进行图谱重新构建
4. 支持通过图谱搜索接口自定义对接搜索应用

### 附录：Dolphin-lang 编排语言

#### C.1 概述

Dolphin-lang（DolphinLanguage）是爱数自研的一种专为 AI 工作流程设计的编程语言，专注于指令调优和对话能力增强，支持 DeepSeek 等多种基座模型。Dolphin-lang创新性地融合了编程语言的精确控制与自然语言的易用性。

#### C.2 核心技术特性

**数据处理与变量系统：**
- 多样化的变量定义方式：使用`->`操作符进行基础赋值操作，`>>`操作符实现内容追加操作
- 丰富的变量访问语法：通过`$变量名`直接访问基础变量，使用`$变量名[索引]`访问列表元素，或通过`$变量名.属性.子属性`的链式语法访问嵌套数据结构
- 复合数据结构支持：支持处理字符串、列表、字典等复杂数据结构

**程序流程控制能力：**
- 循环迭代结构：通过`/for/`结构实现对各类集合数据的遍历处理
- 条件分支结构：采用`/if/`、`elif`、`else`层级结构实现多条件判断
- 控制结构嵌套：支持控制结构的嵌套组合

**AI模型调用能力：**
- 模型调用语法：通过`/prompt/`指令直接调用大模型
- 结果存储机制：模型输出内容能够无缝存储至变量
- 上下文感知：支持完整的上下文传递机制
- 模型响应模式：支持返回最终完整结果和实时流式返回两种模式

**工具调用与集成能力：**
- 函数式调用语法：通过`@工具名（参数列表）`的简洁语法直接调用外部工具
- 参数传递机制：支持将多种类型参数传递给工具
- 模块化管理：通过import语句实现

**智能决策与执行模式：**
- 判断模式：`/judge/`指令实现任务类型的自动判别和最优执行路径的选择
- 探索模式：`/explore/`模式提供更灵活的交互式执行方式，允许模型在思考过程中动态决定是否需要调用工具以及调用哪些工具
- 结构化输出：执行结果以结构化形式返回，包含思考过程、中间结果等详细信息
